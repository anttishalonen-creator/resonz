Resonz v1 implementation-ready product + engineering spec
Document purpose
This document is intended to be handed directly to a coding agent or developer as the primary source of truth for building Resonz v1.
It defines: - product scope - frozen v1 decisions - UI structure - interaction rules - state model - audio-engine behavior - watch-sync behavior - persistence - module structure - acceptance criteria
This spec favors clarity, minimalism, and ship-ability over overengineering.
________________


1. Product summary
1.1 Product name
Resonz
1.2 Product concept
Resonz is a minimalist Android app for guided binaural-beat sessions.
The app has: - four intent presets - one main frequency control - three fine-tuning controls - optional watch-based adaptive guidance
The feel should be: - clean - modern - clinical - calm - trustworthy
It must not feel like: - a mystical chakra app - a marketplace of tracks - a DJ tool - a medical device
1.3 Primary user promise
User can: 1. tap a preset 2. see the controls animate into place 3. fine-tune the session 4. press Play 5. get a smooth, high-quality session
________________


2. Frozen v1 decisions
These are considered fixed unless explicitly changed.
2.1 Main control hierarchy
* Beat is the hero slider and the most visually prominent control.
* Drift, Time, and Tone are grouped underneath as a fine-tuning cluster.
* Watch Sync is a separate toggle row under the fine-tuning cluster.
2.2 Presets
The main screen contains exactly four preset buttons: - Deep Sleep - Drift to Sleep - Focus - Earth Sync
2.3 Preset behavior
Pressing a preset button: - visually activates the button - animates all sliders/toggle targets into place - updates the audio engine targets smoothly - does not auto-start playback
2.4 Audio implementation direction
v1 should use real-time synthesis, not a pre-rendered-track-only architecture.
Reason: - preset animation must map to real parameter change - Beat must be live and meaningful - Drift must evolve in-session - Tone must change sound character in real time - Watch Sync must gently influence session progression
2.5 Design direction
Use the same broad visual family as Cognisenz: - white / very light gray background - dark navy active elements - pale blue-gray inactive controls - minimal typography - generous whitespace - subtle borders / shadows only

4. Core UX model
4.1 User mental model
Resonz is a preset-first system with live manual refinement.
The user understands the app as: - choose a mode - adjust one big frequency slider - tweak three smaller controls - optionally allow watch guidance - play
4.2 Control roles
Beat
Main target binaural beat frequency.
Drift
How much the beat travels or evolves over the session.
Time
Session duration. Also affects how fast Drift unfolds.
Tone
How warm / bright the sound character feels.
Watch Sync
Allows the app to adapt the session path using heart-rate trend.
4.3 Important principle
The app should feel like: - a calm instrument - not a spreadsheet of settings
Therefore: - show few controls - hide technical complexity behind simple labels - keep the default state beautiful and usable

8. Main screen detailed spec
8.1 Purpose
Main screen is where the user configures the session.
8.2 Layout order
From top to bottom: 1. logo mark 2. wordmark 3. preset button grid 4. hero Beat slider section 5. grouped fine-tuning section 6. Watch Sync row 7. bottom action buttons
8.3 Preset button grid
Use a 2x2 grid.
Row 1: - Deep Sleep - Drift to Sleep
Row 2: - Focus - Earth Sync
Preset button style
Inactive: - pale tile background - dark navy text - subtle border / no heavy shadow
Active: - filled navy - white text - subtle active elevation
Preset button dimensions
* height: 56dp
* full width in each half-column
* corner radius: 8dp
Preset button interaction
Tap behavior: - animate active-state visuals - animate all control values to preset targets over 350ms - update audio targets with smoothing - if playback already active, update sound live without stopping session
Long press behavior: none in v1.
8.4 Hero Beat section
Structure
Row 1: - left label: BEAT - right live value: e.g. 8.0 Hz
Row 2: - large slider rail
Row 3: - zone labels below rail: DEEP, DRIFT, CALM, FOCUS
Beat slider requirements
* most prominent control on the page
* larger thumb than other sliders
* slider should feel smooth and premium
Beat value formatting
* one decimal place when >= 1.0
* two decimals when < 1.0
* examples:
   * 3.0 Hz
   * 7.8 Hz
   * 0.75 Hz
Beat slider scale
Use a non-linear mapping so low frequencies are easier to control.
Recommended mapping: - UI position p in [0,1] - beatHz = exponential interpolation between min and max
Recommended bounds: - min: 0.5 Hz - max: 40.0 Hz
Reason: A linear scale would waste too much travel on the high end and make the low end too cramped.
Zone labels for user guidance
These are visual guidance only, not hard mode locks: - DEEP = 0.5–4.0 Hz - DRIFT = 4.0–8.0 Hz - CALM = 8.0–12.0 Hz - FOCUS = 12.0–40.0 Hz
8.5 Fine-tuning cluster
Under the Beat section, place the three fine-tuning controls visually closer together.
Order: 1. Drift 2. Time 3. Tone
These should be grouped more tightly than the Beat section.

9. Fine-tuning controls spec
9.1 Drift control
Purpose
Controls how much the beat evolves over the session.
UI representation
Row 1: - left label: DRIFT - right live value text: Off, Low, Medium, or High
Row 2: - slider rail with 4 snap positions
Allowed values
Discrete positions only: - 0 = Off - 1 = Low - 2 = Medium - 3 = High
Internal normalized value
Also expose normalized float for engine: - Off = 0.00 - Low = 0.33 - Medium = 0.66 - High = 1.00
UX notes
* snapping should feel intentional, not wobbly
* when preset changes, slider thumb animates to target position
9.2 Time control
Purpose
Defines session duration and indirectly affects drift rate.
UI representation
Time should appear as a slider-like control with snap points.
Row 1: - left label: TIME - right live value text: e.g. 45 Min, 90 Min, All Night
Row 2: - slider rail with snap points
Allowed values for v1
Discrete positions: - 20 Min - 30 Min - 45 Min - 90 Min - All Night
Internal values
* 20m = 1200 sec
* 30m = 1800 sec
* 45m = 2700 sec
* 90m = 5400 sec
* All Night = 28800 sec (8 hours default internal target)
UX note
Although user suggested a slider, the implementation should be a snapped slider / segmented rail hybrid so it still feels precise.
9.3 Tone control
Purpose
Controls sound character, not the binaural beat itself.
UI representation
Row 1: - left label: TONE - right live value text: Warm, Neutral, or Bright
Row 2: - continuous slider from Warm to Bright
Row 3: - small labels under rail: WARM on left, BRIGHT on right
Allowed user range
Continuous 0.0–1.0
UI live label mapping
* 0.00–0.33 => Warm
* 0.34–0.66 => Neutral
* 0.67–1.00 => Bright
Internal meaning
Tone changes: - carrier center frequency - harmonic mix - filtering / brightness of sound - optional body-layer strength
Tone does not change: - beat frequency - duration - preset identity

15. What happens when a preset is selected
15.1 UI logic
When preset button P is tapped: 1. store selectedPreset = P 2. store isModified = false 3. animate visible controls to preset target values 4. if playback active, call audio-engine target update 5. if playback inactive, update preview state only
15.2 Animation timing
Recommended: - control animation duration: 350ms - easing: standard ease-in-out - toggle change: 200ms
15.3 Audio transition timing
Audio should move more slowly than UI so it feels calm.
Recommended parameter smoothing: - Beat target smoothing: 500–1200ms - Tone target smoothing: 600–1500ms - preset full transition envelope: 1–3 sec, depending on parameter distance
________________


16. What happens when user tweaks sliders after preset selection
16.1 Modification rules
If user changes any control away from preset target by meaningful threshold: - keep the last preset button visually active - set isModified = true - show small helper text under preset grid: Modified
This is simpler than adding a separate CUSTOM button in v1.
16.2 Meaningful thresholds
Use these thresholds to decide whether config counts as modified: - Beat differs by > 0.15 Hz - Drift differs by one snap level - Time differs by one snap level - Tone differs by > 0.08 - Watch Sync differs from preset default

17. Audio-engine product spec
17.1 Primary requirement
Audio must feel smooth, pleasant, and non-haunting.
17.2 Non-goals
Do not ship raw naked sine tones as the default final sound.
Even if the engine internally begins from sine-based carriers, the audible result should be softened and pleasant.
17.3 Core audio layers
v1 recommended synthesis stack: 1. Binaural carrier layer 2. Body layer for warmth and thickness 3. Very subtle air / noise layer if needed 4. Master output envelope + limiter
Layer 1: binaural carrier layer
Produces the actual left/right frequency difference.
Layer 2: body layer
Adds warmth and reduces the empty haunted feeling.
Layer 3: subtle air/noise layer
Optional low-level smooth noise to soften harshness.
Layer 4: master stage
Ensures click-free fades and clipping protection.
17.4 Audio quality rules
* no clicks on start / stop
* no clicks on slider change
* no abrupt phase reset during normal parameter changes
* avoid clipping
* avoid brittle or shrill sound at higher tone values
* maintain stereo integrity
* support headphone playback cleanly
17.5 Output format
Recommended internal target: - stereo - 48 kHz preferred if supported, otherwise 44.1 kHz fallback - float internally if convenient, PCM16 acceptable at output boundary
17.6 Session start behavior
On session start: - initialize engine with current state - fade in over 400–1200ms - do not blast full amplitude instantly
17.7 Session end behavior
On stop / end: - fade out over 800–2000ms - stop engine after fade completes
17.8 Parameter smoothing
Every user-controlled parameter must be smoothed.
Parameters requiring smoothing: - beatHz - carrierCenter - toneBrightness - body mix - master gain
Recommended smoothing type: - one-pole smoothing or ramp interpolation

18. Audio-engine technical behavior
18.1 Fundamental binaural model
At any time: - left frequency = carrierCenter - beatHz/2 - right frequency = carrierCenter + beatHz/2
The brain perceives the difference as the binaural beat.
18.2 Carrier center selection
Carrier center is influenced by Tone.
Suggested mapping: - Warm => lower carrier center - Neutral => mid carrier center - Bright => higher carrier center
Recommended base range: - warm area: 140–180 Hz - neutral area: 180–240 Hz - bright area: 240–320 Hz
18.3 Tone mapping suggestion
toneNormalized in [0,1]
Map to: - carrier center - harmonic emphasis - filter brightness
Example conceptual mapping: - 0.0 = warm, lower center, stronger body layer, softer highs - 0.5 = neutral - 1.0 = brighter, higher center, lighter body
18.4 Body layer suggestion
Use a subtle, musically pleasant under-layer that follows tone choice.
Requirements: - never dominate the binaural layer - never sound like an obvious melody in v1 default sound - should mainly add warmth and fullness
18.5 Optional noise layer suggestion
Very subtle smooth noise may be used to reduce sterile emptiness.
Requirements: - shared between channels or carefully managed - must not destroy the binaural effect - keep level low

19. Drift engine behavior
19.1 Concept
Beat slider defines the anchor. Drift defines how much movement happens over the session. Time defines how fast that movement unfolds. Preset defines the profile type.
19.2 Profile types
Use one of these profile types internally: - STABLE - DESCEND_SOFT - DESCEND_ACTIVE - GENTLE_AROUND_ANCHOR - FOCUS_STABLE
19.3 Recommended mapping by preset
* Deep Sleep => DESCEND_SOFT
* Drift to Sleep => DESCEND_ACTIVE
* Focus => FOCUS_STABLE
* Earth Sync => GENTLE_AROUND_ANCHOR
19.4 Drift behavior by profile
STABLE
* keep beat near anchor
* very small movement only
DESCEND_SOFT
* gradually move toward a lower target over time
* movement mild even at Low drift
DESCEND_ACTIVE
* more noticeable downward journey over time
* higher drift means larger total descent
GENTLE_AROUND_ANCHOR
* remain near anchor with tiny movement / breathing feel
* no major descent unless user sets high drift and long time
FOCUS_STABLE
* keep beat effectively fixed
* optional tiny early-settling ramp only
19.5 Time + Drift interaction
This is the key rule.
* Short Time + High Drift => faster movement
* Long Time + High Drift => slower broader movement
* Drift Off => stable regardless of time
19.6 Recommended v1 descent rule
For descend-type profiles, compute a target end beat based on: - anchor beat - drift amount - lower floor by mode - time duration
Suggested simple rule for descend presets
For Deep Sleep and Drift to Sleep: - start at anchor beat - compute max downward distance based on drift - map time into curve duration - use a smooth monotonic curve, not staircase jumps
Example conceptual floors: - Deep Sleep floor: 1.5 Hz - Drift to Sleep floor: 2.0 Hz
Example drift distances
For Deep Sleep with anchor 3.0: - Off => end 3.0 - Low => end 2.5 - Medium => end 2.0 - High => end 1.5
For Drift to Sleep with anchor 8.0: - Off => end 8.0 - Low => end 6.0 - Medium => end 4.0 - High => end 2.5
These are defaults. Agent may refine the curves if sound behavior is smoother.
19.7 Update cadence
Engine may recalculate target path continuously or in small intervals.
User should perceive: - a continuous gentle drift - never discrete stepping

20. Watch Sync behavior spec
20.1 Product principle
Watch Sync should guide the session, not directly control it.
20.2 Explicit rule
Do not map current heart rate directly to beat frequency.
Bad example: - HR 72 bpm => 1.2 Hz beat
Do not do this.
20.3 Correct behavior
Use heart-rate trend to decide whether the session should: - hold at a lighter stage longer - descend more quickly - pause descent temporarily - remain stable
20.4 Watch Sync inputs
Collect these values from watch side: - timestamp - heartRateBpm - accuracy / confidence if available - source status
20.5 Phone-side adaptive logic
Phone should compute: - baseline heart rate from session start window - smoothed current heart rate - trend delta vs baseline
Recommended baseline window: - first 60–120 sec of session, once enough valid samples exist
Recommended smoothing: - rolling average or EWMA to suppress jitter
20.6 Adaptation rules for sleep presets
For supported sleep presets: - if HR remains above baseline by threshold, hold slightly higher / lighter longer - if HR drops below baseline or clearly trends downward, allow descent - max adjustment per evaluation should remain subtle
Suggested thresholds
These are reasonable starting values, not sacred truths: - elevated threshold: +4 bpm versus smoothed baseline - settled threshold: -3 bpm or stronger downward trend
Suggested evaluation interval
* every 15–30 sec
Suggested max beat adjustment from watch influence
* no more than about 0.2–0.5 Hz per evaluation
* never enough to create sudden audible jumps
20.7 Focus-mode rule
In v1, Focus does not use Watch Sync.
Reason: - simpler implementation - avoids overpromising - sleep logic is more intuitive for HR trend adaptation
20.8 Earth Sync rule
Earth Sync may optionally use a very gentle hold / release adaptation if implemented, otherwise leave disabled in v1.
20.9 Failure behavior
If watch disconnects mid-session: - keep current session running normally - freeze Watch Sync adaptation - show muted status Watch disconnected - do not interrupt audio

55. Non-negotiable coding rules for agent
1. Do not hardcode random dimensions inside feature composables.
2. Do not hardcode random hex colors inside screen files.
3. All layout values must come from design-token files or dedicated constants.
4. All preset values must come from PresetDefaults.kt.
5. All slider-to-domain mappings must live in domain files, not in composables.
6. MainScreen must stay visually dumb; logic belongs in ViewModel/domain.
7. Audio engine must be controlled through an interface.
8. Watch sync must fail gracefully without breaking sessions.
9. Keep Kotlin files focused; do not create giant 1000-line files if avoidable.
10. Preserve the locked UI structure and spacing from this spec.
________________


56. Immediate first implementation note
The first code pass should not start with the watch or with audio perfection.
It should start with: - clean Compose layout - state model - preset-to-control animation - value mapping utilities - preview states
Once the UI state behavior feels correct, connect the audio engine.
That is the fastest path to a working Resonz v1.
